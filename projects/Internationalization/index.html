The real problem

Translation was not the hard part. Control was.

The platform already allowed teams to create customer communication templates. The challenge was that multilingual content was becoming a product and architecture problem, not just a copywriting problem. Teams needed localized messages, but the platform also needed consistency, reviewability, publishing discipline, and a way to keep variants tied to one logical template.

1 Logical template needed to support multiple language variants.
Many Teams needed a repeatable model instead of custom work for each use case.
Zero Room for accidental mixed-language messages in customer-facing communication.
Clear Ownership, review, preview, and publishing rules had to be designed up front.

Why it mattered

A communication template is small on screen, but large in consequence. If language handling is messy, the customer sees the mess immediately.

Internationalization work can look deceptively simple from the outside. Add a language selector. Store translated text. Show the right version. Done.

In a real platform, that view breaks quickly. A template may have dynamic values, required legal text, reusable components, preview modes, approvals, publishing rules, channel constraints, and downstream delivery expectations. Once every one of those pieces needs to behave correctly across languages, the project becomes a design problem.

My contribution was to prepare the technical design document and shape the implementation direction. I documented the current limitation, compared solution approaches, drove the workflow thinking, and helped convert a broad product ask into a platform design that engineers and product stakeholders could discuss, estimate, and build.

The public version of that design is simple: keep the authoring model understandable, keep the localized variants connected, and make it difficult for teams to publish an incomplete language experience by accident.

Design direction

Two paths were possible. Only one scaled well.

The core design decision was whether each language should become an independent template or whether a single template should own multiple localized variants. Both models can work, but they create very different long-term behavior.

Rejected path

Separate template per language

This approach is easier to understand initially, but it creates duplicate maintenance, fragmented ownership, inconsistent publishing, and more chances for variants to drift apart over time.

Chosen path

One template with localized variants

This model keeps all language versions tied to one logical template. It supports cleaner previews, stronger governance, and a better authoring experience as more languages are introduced.

The workflow had to feel calm.

The product should not make authors think like database designers. The interface needed a predictable flow: create the base template, add a localized version, review missing content, preview the customer experience, and publish only when the required pieces are ready.

That calmness was the real architecture goal. Behind the scenes, the system could enforce structure. On the screen, the user should feel guided.

Architecture thinking

The cleanest UI was backed by strict mental models.

The public-safe architecture can be described with four ideas: stable template identity, localized content variants, validation before publishing, and consistent rendering across communication channels.

Public-safe system map

Template identity One logical template remains the source of structure and ownership.
Localized variants Language-specific content is attached as a variant, not scattered as unrelated templates.
Authoring checks Required fields, placeholders, and preview states are validated before release.
Delivery readiness The selected language version is prepared for customer-facing channels in a consistent way.

Keep variants connected

A localized version should never feel like a forgotten copy. It should stay attached to the template it represents.

Block incomplete states

The safest multilingual system is one that makes invalid states difficult to publish.

Design for the next language

The first rollout should teach the platform how to support the second, third, and fourth rollout more gracefully.

Conclusion

Internationalization is empathy disguised as architecture.

The most interesting part of this project was not adding another language. It was designing respect into the platform. Respect for customers who should receive communication in a language they are comfortable with. Respect for authors who need a clear workflow. Respect for engineers who need a maintainable model. And respect for the business, which needs global capability without global chaos.

That is why this project stays with me. It reminded me that a good technical design document is more than a document. It is a bridge between an unclear need and a system that many people can trust.
